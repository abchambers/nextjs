# First Classroom Pilot

## Objective

Prove that Frontline Forecast helps a real instructor teach forecasting practice more effectively than a collection of disconnected weather websites, spreadsheets, and ad hoc feedback.

## Pilot shape

| Element | Recommendation |
| --- | --- |
| Customer | One instructor and one course section |
| Size | 5–20 students for the first cohort |
| Duration | 3–6 weeks or one focused forecasting unit |
| Access | Guided, no broad self-serve enrollment yet |
| Cost | Free or clearly documented pilot pricing; no ambiguous promises |
| Support | One direct instructor contact and a short feedback cadence |

## Pilot entry criteria

- The production registration and confirmation flow works end-to-end.
- The instructor can create a classroom and a multi-day assignment.
- A student can join the intended workspace and submit a linked forecast.
- The instructor can see the class assignment, review a submission, and leave feedback.
- The class outlook is demonstrably private to the classroom.
- Provider/source status and user support contact are known.

## Instructor onboarding

1. Define the course purpose, target class size, date range, and forecast skill being practiced.
2. Create the school organization, instructor membership, and classroom.
3. Walk through one sample assignment and one sample student submission.
4. Agree on how student work is graded, what remains private, and how support issues are reported.
5. Distribute time-limited invitations only after one student test account works.

## Success metrics

Measure outcomes, not clicks alone.

| Measure | Evidence |
| --- | --- |
| Activation | Instructor creates a class and at least one assignment without developer intervention |
| Student completion | Students who register, join, and submit each assigned day |
| Review completion | Submissions with instructor feedback or a saved grade |
| Return use | Instructor creates a second assignment; students return for the next cycle |
| Workflow clarity | Support requests categorized by confusing step or failure |
| Product value | Instructor interview: what replaced existing work, what still required outside tools |
| Operating cost | Requests/usage for radar, models, archives, and map tiles per active student |

## Weekly cadence

- **Before class:** Confirm source availability and assignment setup.
- **During class:** Collect only essential support issues; do not alter workflows mid-assignment unless it blocks learning.
- **After assignment:** Review completion, feedback use, errors, and cost.
- **Weekly instructor check-in:** 15 minutes: what was useful, confusing, missing, or redundant?
- **Pilot closeout:** Summarize results, requested changes, price sensitivity, and renewal conditions.

## Pilot exit decision

Advance to paid/school licensing only when the instructor independently completes the core loop and can name a clear reason to use the product again. Otherwise, improve the workflow before adding billing, integrations, or a public sales funnel.
